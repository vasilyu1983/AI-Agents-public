# Data Ingestion Patterns

## Tool Comparison: dlt vs Airbyte

| Feature | dlt | Airbyte |
|---------|-----|---------|
| **Approach** | Python library | Platform (Docker) |
| **Setup** | `pip install dlt` | Docker Compose / K8s |
| **Configuration** | Python code | YAML / UI |
| **Connectors** | Code-first + templates | Managed connector catalog |
| **Custom sources** | Python decorators | Low-code CDK |
| **Destinations** | ClickHouse, DuckDB, Snowflake, BigQuery, Postgres | 50+ destinations |
| **Incremental** | Built-in | Built-in |
| **Schema evolution** | Automatic | Automatic |
| **AI assistance** | Works well with code review | Works well for config generation |
| **Best for** | Python teams, AI-assisted development | Complex sources, GUI preference |

## AI-Assisted Pipeline Creation (Use Carefully)

AI tools are useful for scaffolding connectors and configs, but treat generated ingestion code as untrusted until it passes real backfill/replay tests.

### AI-Generated Pipeline Example

```python
# Generated by Claude/Cursor with human review
import dlt
from dlt.sources.rest_api import rest_api_source

@dlt.source
def stripe_source(api_key: str = dlt.secrets.value):
    """Stripe data source with incremental loading."""

    config = {
        "client": {
            "base_url": "https://api.stripe.com/v1",
            "auth": {"type": "bearer", "token": api_key},
            "paginator": {
                "type": "cursor",
                "cursor_path": "data[-1].id",
                "cursor_param": "starting_after"
            }
        },
        "resources": [
            {
                "name": "customers",
                "endpoint": {"path": "customers"},
                "write_disposition": "merge",
                "primary_key": "id"
            },
            {
                "name": "invoices",
                "endpoint": {
                    "path": "invoices",
                    "params": {"created[gte]": "{incremental.created}"}
                },
                "incremental": {"cursor_path": "created"},
                "write_disposition": "merge",
                "primary_key": "id"
            },
            {
                "name": "subscriptions",
                "endpoint": {"path": "subscriptions"},
                "write_disposition": "merge",
                "primary_key": "id"
            }
        ]
    }

    return rest_api_source(config)

# Run pipeline
if __name__ == "__main__":
    pipeline = dlt.pipeline(
        pipeline_name="stripe_to_clickhouse",
        destination="clickhouse",
        dataset_name="stripe_raw"
    )
    load_info = pipeline.run(stripe_source())
    print(load_info)
```

### Best Practices for AI-Assisted Development

| Do | Don't |
|----|-------|
| Start with verified dlt sources as templates | Generate from scratch for common APIs |
| Review AI-generated pagination logic | Trust complex auth handling blindly |
| Test with small datasets first | Deploy without incremental testing |
| Use dlt's built-in schema contracts | Skip data validation |
| Version control AI-generated code | Treat as disposable |

## dlt Patterns

### Basic Pipeline

```python
import dlt

@dlt.source
def my_source():
    @dlt.resource(write_disposition="merge", primary_key="id")
    def users():
        for page in paginate_api("/users"):
            yield page

    return users

# Run pipeline
pipeline = dlt.pipeline(
    pipeline_name="my_pipeline",
    destination="clickhouse",
    dataset_name="raw"
)
pipeline.run(my_source())
```

### Incremental Loading

```python
@dlt.resource(
    write_disposition="merge",
    primary_key="id"
)
def orders(
    updated_at=dlt.sources.incremental("updated_at", initial_value="2024-01-01")
):
    """Load orders incrementally by updated_at"""
    for page in api.get_orders(since=updated_at.last_value):
        yield page
```

### REST API Source

```python
from dlt.sources.rest_api import rest_api_source

config = {
    "client": {
        "base_url": "https://api.example.com/v1",
        "auth": {"type": "bearer", "token": dlt.secrets["api_token"]}
    },
    "resources": [
        {
            "name": "users",
            "endpoint": {"path": "users", "paginator": "json_link"},
            "write_disposition": "merge",
            "primary_key": "id"
        },
        {
            "name": "orders",
            "endpoint": {
                "path": "orders",
                "params": {"updated_since": "{incremental.updated_at}"}
            },
            "incremental": {"cursor_path": "updated_at"}
        }
    ]
}

source = rest_api_source(config)
```

### Database Replication

```python
from dlt.sources.sql_database import sql_database

# Full table replication
source = sql_database(
    credentials="postgresql://user:pass@host:5432/db",
    schema="public",
    table_names=["users", "orders", "products"]
)

# Incremental with merge
source = sql_database(
    credentials="postgresql://user:pass@host:5432/db",
    table_names=["orders"],
    incremental=dlt.sources.incremental("updated_at")
)
```

### Destination: ClickHouse

```python
pipeline = dlt.pipeline(
    pipeline_name="to_clickhouse",
    destination=dlt.destinations.clickhouse(
        credentials="clickhouse://user:pass@host:9000/db"
    ),
    dataset_name="raw"
)

# ClickHouse-specific settings
pipeline.run(
    source,
    table_format="delta",  # or "iceberg"
    loader_file_format="parquet"
)
```

## Airbyte Patterns

### Connection Configuration (YAML)

```yaml
# airbyte/connections/postgres_to_clickhouse.yaml
sourceDefinitionId: postgres
destinationDefinitionId: clickhouse

source:
  host: source-db.example.com
  port: 5432
  database: production
  username: ${POSTGRES_USER}
  password: ${POSTGRES_PASSWORD}
  ssl_mode: require
  replication_method:
    method: CDC
    publication: airbyte_publication
    replication_slot: airbyte_slot

destination:
  host: clickhouse.example.com
  port: 8443
  database: raw
  username: ${CLICKHOUSE_USER}
  password: ${CLICKHOUSE_PASSWORD}
  ssl: true

streams:
  - name: users
    sync_mode: incremental
    destination_sync_mode: append_dedup
    cursor_field: updated_at
    primary_key: [id]

  - name: orders
    sync_mode: incremental
    destination_sync_mode: append_dedup
    cursor_field: updated_at
    primary_key: [id]

schedule:
  scheduleType: cron
  cronExpression: "0 */6 * * *"  # Every 6 hours
```

### Custom Connector (Low-Code CDK)

```yaml
# airbyte/sources/my_api/manifest.yaml
version: 0.79.0

type: DeclarativeSource
definitions:
  selector:
    type: RecordSelector
    extractor:
      type: DpathExtractor
      field_path: ["data"]

  requester:
    type: HttpRequester
    url_base: "https://api.example.com/v1"
    http_method: GET
    authenticator:
      type: BearerAuthenticator
      api_token: "{{ config['api_key'] }}"

  paginator:
    type: DefaultPaginator
    page_token_option:
      type: RequestOption
      inject_into: request_parameter
      field_name: cursor
    pagination_strategy:
      type: CursorPagination
      cursor_value: "{{ response.next_cursor }}"

streams:
  - type: DeclarativeStream
    name: users
    primary_key: id
    retriever:
      type: SimpleRetriever
      requester: "*ref(definitions.requester)"
      record_selector: "*ref(definitions.selector)"
      paginator: "*ref(definitions.paginator)"
    $parameters:
      path: "/users"

spec:
  type: Spec
  connection_specification:
    type: object
    required: [api_key]
    properties:
      api_key:
        type: string
        airbyte_secret: true
```

## Incremental Loading Strategies

### Strategy 1: Timestamp-Based

```python
# Best for: Tables with updated_at column
@dlt.resource(write_disposition="merge", primary_key="id")
def orders(updated_at=dlt.sources.incremental("updated_at")):
    yield from api.get_orders(since=updated_at.last_value)
```

### Strategy 2: ID-Based

```python
# Best for: Append-only tables with auto-increment ID
@dlt.resource(write_disposition="append")
def events(last_id=dlt.sources.incremental("id", initial_value=0)):
    yield from api.get_events(after_id=last_id.last_value)
```

### Strategy 3: Cursor-Based (API Pagination)

```python
# Best for: APIs with cursor pagination
@dlt.resource
def items(cursor=dlt.sources.incremental("cursor")):
    while True:
        response = api.get_items(cursor=cursor.last_value)
        yield response["items"]
        if not response.get("next_cursor"):
            break
        cursor.last_value = response["next_cursor"]
```

### Strategy 4: Full Refresh with Dedup

```python
# Best for: Small tables, no reliable incremental column
@dlt.resource(write_disposition="replace")
def config_table():
    yield from api.get_all_config()
```

## Error Handling

### dlt Retry Configuration

```python
import dlt
from dlt.sources.helpers.rest_client import RESTClient

client = RESTClient(
    base_url="https://api.example.com",
    request_timeout=30,
    retry_count=5,
    retry_wait=2,
    retry_on_status_codes=[429, 500, 502, 503, 504]
)
```

### Dead Letter Queue Pattern

```python
@dlt.resource
def resilient_source():
    for record in source_records():
        try:
            validated = validate(record)
            yield validated
        except ValidationError as e:
            # Send to DLQ
            yield dlt.mark.with_table_name(
                {"record": record, "error": str(e)},
                "dead_letter_queue"
            )
```

## Best Practices

1. **Always use incremental loading** when possible
2. **Set primary keys** for merge operations
3. **Use schema contracts** to catch breaking changes
4. **Monitor pipeline state** (dlt stores state automatically)
5. **Test with small datasets** before full loads
6. **Use secrets management** (dlt.secrets, environment variables)
7. **Implement idempotency** for rerunnable pipelines
