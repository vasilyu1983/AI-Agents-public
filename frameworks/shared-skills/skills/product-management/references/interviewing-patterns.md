# 1. Interview Structure Templates

## 1.1 Standard Discovery Interview (15–30 minutes)

Intro (1 min)
Thank them
Get permission to record (optional)
Set expectation: “No pitching today”
Context Trigger (2–3 min)
“Tell me about the last time you…”
“What were you trying to do?”
Deep Dive (10–15 min)
“Walk me through it step-by-step”
“What did you do first?”
“Then what happened?”
“What was hardest?”
“What did you do next?”
Alternatives (5 min)
“What other tools or methods did you try?”
“What worked well / poorly?”
Impact (3 min)
“What happens if you don’t solve this?”
“How often does this occur?”
“What’s the cost or consequence?”
Closing (2–3 min)
Commitment test
Ask for artifacts (screenshots, spreadsheets)
Ask who else to talk to
Thank them

# 2. Question Patterns

## 2.1 “Mom Test” Patterns (Good Questions)

Use for ANY discovery interview.

**Behavior-focused**

- “Tell me about the last time you…”
- “Walk me through how you solved that.”
- “What did you do right before / after?”

**Depth probes**

- “Why was that important?”
- “What made that hard?”
- “How do you measure success today?”

**Alternatives**

- “What else did you consider?”
- “Why not use X?”
- “What frustrated you about Y?”

**Impact**

- “How did that affect you?”
- “What happens if nothing changes?”

---

## 2.2 Questions to Avoid (Bad Patterns)

- “Would you use this?”  
- “Do you like this idea?”  
- “How much would you pay?”  
- “Is this a good idea?”  
- Hypotheticals (“Imagine you had…”)  

---

# 3. Commitment Test Patterns

Use to separate polite interest from real demand.

## 3.1 Ask for Time

- “Can we schedule a follow-up to review a prototype?”
- **Pass if:** they book time  
- **Fail if:** vague or postpones

## 3.2 Ask for Risk

- “Will you share anonymized data so we can test?”
- **Pass if:** they send data  
- **Fail if:** excuses or delays

## 3.3 Ask for Money (if applicable)

- “Would you pre-pay for early access?”
- **Pass if:** real payment  
- **Fail if:** “circle back later”

## 3.4 Ask for Referrals

- “Who else should we speak with?”
- **Pass:** they introduce  
- **Fail:** they decline

# 4. Signal Strength Scoring

Use after each interview.

Signal Strength:
Strong:

- Multiple examples
- High-frequency problem
- Severe pain / high cost
- Clear workaround that is painful
Medium:
- Some examples but inconsistent
Weak:
- Anecdotal, hypothetical, or vague

# 6. Participant Selection Patterns

## 6.1 Who to Interview

- Current users experiencing the problem  
- Evaluators / prospects  
- Recent churn  
- Heavy users / power users  
- “Aspirational” segments (if early stage)

## 6.2 Sample Size Targets

- 5–8 per segment  
- Stop when patterns repeat (“saturation”)

# 7. Interview Logistics

## 7.1 Scheduling Checklist

- [ ] Clear invite purpose  
- [ ] Recording permission  
- [ ] 30-minute slot preferred  
- [ ] Agenda shared (optional)

## 7.2 Setup

- [ ] Quiet environment  
- [ ] Interviewer camera on  
- [ ] Notetaker optional but useful  
- [ ] Prototype/tools ready (if testing usability)

# 8. Interview Types

## 8.1 Problem Interviews (Discovery)

Purpose: understand pains, context, frequency.

Checklist:

- [ ] No solution pitching  
- [ ] Behavior-first questions  
- [ ] Capture quotes  
- [ ] Identify opportunities  

## 8.2 Solution Interviews (Validation)

Purpose: test prototype or value proposition.

Checklist:

- [ ] Validate assumptions  
- [ ] Walkthrough tasks  
- [ ] Test willingness to pay / adopt  
- [ ] Observe confusion, hesitation  

## 8.3 Usability Interviews

Purpose: find friction in the product.

Checklist:

- [ ] Give task, not instructions  
- [ ] Let them struggle  
- [ ] Capture task completion + errors

# 9. Decision Trees

## 9.1 Is This a Good Interview Candidate?

Has this person experienced the problem?
├─ No → Skip
└─ Yes
↓
Can they recall a recent event?
├─ No → Low value
└─ Yes → High value

---

## 9.2 Should You Schedule a Follow-Up?

Did they show strong evidence of recurring pain?
├─ No → Do not schedule
└─ Yes
↓
Did they offer artifacts / data?
├─ Yes → High-signal → schedule
└─ No → Medium-signal → optional

---

# 10. Validation Interview Pattern

Use this when testing a prototype or concept.

Reconfirm the problem
Show prototype (no explanation)
Give task: “Show me how you’d complete…”
Capture:
Confusion points
Incorrect paths
Questions asked
Ask for commitment:
Follow-up?
Early access?
Data sharing?

---

# 11. Anti-Patterns

- AVOID: Talking more than the participant  
- AVOID: Leading questions (“Wouldn’t it be great if…?”)  
- AVOID: Jumping to features  
- AVOID: Taking notes instead of quotes  
- AVOID: Asking for opinions rather than facts  
- AVOID: Interviewing random people (not the target segment)  

---

# 12. Definition of Done (Interview Insight)

An interview yields **valid insights** when:

- [ ] At least 1 concrete story captured  
- [ ] Pain points include frequency + severity  
- [ ] Alternatives documented  
- [ ] Impact quantified (time/money/frustration)  
- [ ] Opportunity restated in problem language  
- [ ] Signal strength assigned  
- [ ] Actionable next test identified
